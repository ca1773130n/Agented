  <name>Task 1: Create specialized bot API routes and frontend API client</name>
  <files>backend/app/routes/specialized_bots.py, backend/app/routes/__init__.py, frontend/src/services/api/specialized-bots.ts, frontend/src/services/api/index.ts, frontend/src/services/api/types.ts</files>
  <action>
    1. Create `backend/app/routes/specialized_bots.py`:
    - Blueprint: `specialized_bots_bp = APIBlueprint("specialized_bots", __name__, url_prefix="/admin", abp_tags=[tag])` with tag "specialized-bots"
    - `GET /admin/specialized-bots/status`: Returns the status of all 7 specialized bots. For each bot:
      - `id`: trigger ID (e.g., "bot-vuln-scan")
      - `name`: trigger name
      - `trigger_exists`: bool — check if trigger exists in DB via `get_trigger(bot_id)`
      - `skill_file_exists`: bool — check if `.claude/skills/<slug>/INSTRUCTIONS.md` exists on disk (use the slug derived from the prompt_template command, e.g., "/vulnerability-scan" -> "vulnerability-scan")
      - `trigger_source`: the bot's trigger source (scheduled, github, manual)
      - `enabled`: True if both trigger_exists and skill_file_exists
    - `GET /admin/specialized-bots/health`: Returns health status of external dependencies needed by bots:
      - `gh_authenticated`: bool — result of `SpecializedBotService.check_gh_auth()`
      - `osv_scanner_available`: bool — result of `SpecializedBotService.check_osv_scanner()`
      - `search_index_count`: int — number of documents in FTS5 index via `ExecutionSearchService.get_search_stats()`
    - Import `SpecializedBotService` from `app.services.specialized_bot_service` and `ExecutionSearchService` from `app.services.execution_search_service`. Import trigger DB functions from `app.db`.

    2. Register the blueprint in `backend/app/routes/__init__.py`:
       - Add `from .specialized_bots import specialized_bots_bp` to the imports (alphabetical order).
       - Add `specialized_bots_bp` to the `admin_blueprints` list (the list that applies the 120/minute rate limit via `limiter.limit("120/minute")`). Place it alphabetically among existing entries.
       - Add `app.register_api(specialized_bots_bp)` in the blueprint registration section (alphabetical order).
       All three steps are required — omitting the `admin_blueprints` entry would leave the endpoints unprotected by rate limiting.

    3. Create `frontend/src/services/api/specialized-bots.ts`:
    - Define `specializedBotApi` object following the pattern in existing API modules (e.g., `triggers.ts`):
      - `getStatus()`: GET /admin/specialized-bots/status
      - `getHealth()`: GET /admin/specialized-bots/health
      - `searchLogs(query: string, limit?: number, triggerId?: string)`: GET /admin/execution-search with query params
      - `getSearchStats()`: GET /admin/execution-search/stats
    - All functions use `apiFetch` from `./client`

    4. Add TypeScript types to `frontend/src/services/api/types.ts`:
    - `SpecializedBotStatus`: { id: string, name: string, trigger_exists: boolean, skill_file_exists: boolean, trigger_source: string, enabled: boolean }
    - `SpecializedBotHealth`: { gh_authenticated: boolean, osv_scanner_available: boolean, search_index_count: number }
    - `ExecutionSearchResult`: { execution_id: string, trigger_id: string, trigger_name: string | null, started_at: string, status: string, prompt: string | null, stdout_match: string | null, stderr_match: string | null }
    - `ExecutionSearchResponse`: { results: ExecutionSearchResult[], total: number, query: string }

    5. Re-export `specializedBotApi` from `frontend/src/services/api/index.ts`.

    AVOID: Do not create a separate API module for execution search — include search functions in `specializedBotApi` since it's the same domain. Do not duplicate existing trigger API endpoints.
