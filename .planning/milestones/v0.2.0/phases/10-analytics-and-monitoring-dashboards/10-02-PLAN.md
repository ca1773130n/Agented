    **Health monitor service:**

    Create `backend/app/services/health_monitor_service.py` following MonitoringService pattern:

    ```python
    class HealthMonitorService:
        _job_id = "bot_health_monitoring"
        _cleanup_job_id = "health_alert_cleanup"
        CONSECUTIVE_FAILURE_THRESHOLD = 3
        SLOW_EXECUTION_MULTIPLIER = 3.0
        MISSING_FIRE_MULTIPLIER = 2
    ```

    Methods:
    - `init(cls)` — Register APScheduler job (interval=5 minutes) via SchedulerService.get_scheduler(). Also register daily cleanup job to delete alerts older than 7 days. Follow the `MonitoringService.init()` pattern with `replace_existing=True`.
    - `_check_health(cls)` — Main health check loop. Iterates all triggers. For each trigger:
      a) **Consecutive failures:** Query last N executions from execution_logs. Count consecutive failures from most recent. If count >= CONSECUTIVE_FAILURE_THRESHOLD, create alert with type 'consecutive_failure'. Use a simple Python loop on the last 10 executions (not window functions — keep it simple and fast per 10-RESEARCH.md "health check jobs must be fast <5s").
      b) **Slow executions:** Compute average duration from last 20 executions. If the most recent execution duration exceeds 3x the average, create alert with type 'slow_execution'. Only check if there are at least 5 historical executions for comparison.
      c) **Missing fires (scheduled triggers only):** For triggers with trigger_source='scheduled', check if last_run_at is older than MISSING_FIRE_MULTIPLIER * expected_interval. Expected interval derived from schedule_type: daily=24h, weekly=168h. If exceeded, create alert with type 'missing_fire'. Per 10-RESEARCH.md Pitfall 3: Do NOT apply this to webhook/github triggers.
    - `get_status(cls)` — Returns current health status summary: total_alerts, critical_count, warning_count, last_check_time.

    Create `backend/app/services/report_service.py`:

    ```python
    class ReportService:
        TIME_SAVED_PER_PR_REVIEW_MINUTES = 15
        TIME_SAVED_PER_SECURITY_AUDIT_MINUTES = 30
    ```

    Methods:
    - `generate_weekly_report(cls, team_id=None)` — Generates a team impact report for the last 7 days:
      a) `prs_reviewed` — Count of PR review executions (trigger_source='github') that completed successfully.
      b) `issues_found` — Count of security findings from audit executions.
      c) `estimated_time_saved` — prs_reviewed * TIME_SAVED_PER_PR_REVIEW + audits * TIME_SAVED_PER_SECURITY_AUDIT. Use configurable multiplier per 10-RESEARCH.md Open Question 1.
      d) `top_bots` — Top 5 triggers by execution count in the period.
      e) `bots_needing_attention` — Triggers with >50% failure rate or any active health alerts.
      f) `period_start`, `period_end` — Date range of the report.
    - Query execution_logs and pr_reviews tables directly. Do NOT over-engineer with Jinja2 or PDF generation (10-RESEARCH.md Pitfall 5).

    - `WeeklyReport(BaseModel)`: prs_reviewed, issues_found, estimated_time_saved_minutes, top_bots (list), bots_needing_attention (list), period_start, period_end

    AVOID: Using window functions for consecutive failure detection — a simple reverse-chronological loop over 10 rows is faster and easier to test. AVOID: Alert spam — deduplicate by alert_type+trigger_id within 30 minutes.
