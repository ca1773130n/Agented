<objective>
Implement the backend engine enhancements for conditional logic, edge-aware branching, and approval gates in the workflow DAG execution engine.

Purpose: Enables conditional branching that routes execution to the correct downstream path based on rich expression evaluation on PR attributes (WF-02, WF-05), and adds human-in-the-loop approval gates that pause/resume workflow execution (WF-03). These are the foundational backend changes that all subsequent frontend work depends on.

Output: Enhanced workflow execution service with safe expression evaluator, edge-aware conditional branching, and approval gate node type with persist/resume capability.

Research basis: Research recommends extending the existing `WorkflowExecutionService` and following the proven `threading.Event` approval pattern from `TeamExecutionService._execute_human_in_loop` (07-RESEARCH.md). Expression evaluation uses AST-based safe evaluator to avoid `eval()` code injection (OWASP best practice, CONCERNS.md section 1.4).
</objective>

  <action>
    **Part A: Safe Expression Evaluator**

    Add a `SafeExpressionEvaluator` class (or module-level function) to `workflow_execution_service.py` that evaluates condition expressions against a sandboxed context dict. Based on the research recommendation (07-RESEARCH.md, Recommendation 4), use Python's `ast` module to parse expressions into an AST and walk it, only allowing safe operations.

    Supported operators: `==`, `!=`, `>`, `<`, `>=`, `<=`, `in`, `not in`, `and`, `or`, `not`.
    Supported value types: strings, numbers, booleans, None.
    Supported access: dot-notation attribute access on the context dict (e.g., `pr.lines_changed > 500` where context = `{"pr": {"lines_changed": 600}}`).

    AVOID: Do NOT use `eval()`, `exec()`, `compile()` with exec mode, or `simpleeval`. Build a lightweight evaluator using `ast.parse(expression, mode='eval')` and a recursive `_eval_node()` that handles only `ast.Compare`, `ast.BoolOp`, `ast.UnaryOp`, `ast.Attribute`, `ast.Name`, `ast.Constant`, and `ast.Subscript`. Raise `ValueError` for any unsupported AST node type.

    Implementation pattern:
    ```python
    import ast
    import operator

    SAFE_COMPARE_OPS = {
        ast.Eq: operator.eq, ast.NotEq: operator.ne,
        ast.Gt: operator.gt, ast.Lt: operator.lt,
        ast.GtE: operator.ge, ast.LtE: operator.le,
        ast.In: lambda a, b: a in b, ast.NotIn: lambda a, b: a not in b,
    }

    def evaluate_condition(expression: str, context: dict) -> bool:
        tree = ast.parse(expression, mode='eval')
        return bool(_eval_node(tree.body, context))
    ```

    The evaluator must handle nested dict access: `pr.lines_changed` resolves to `context["pr"]["lines_changed"]`. Support both dot-notation (`pr.branch`) and subscript (`pr["branch"]`).

    **Part B: Enhanced Conditional Node**

    Modify `_execute_conditional_node` to support a new `condition_type: "expression"` mode alongside the existing `has_text`, `exit_code_zero`, `contains` modes. When `condition_type == "expression"`:
    1. Read `expression` from `node_config` (e.g., `"pr.lines_changed > 500 and pr.draft == False"`)
    2. Build context dict from `input_msg.data` (the upstream node's output data)
    3. Call `evaluate_condition(expression, context)` to get a boolean result
    4. Return the existing `WorkflowMessage` with `branch: "true"` or `branch: "false"`

    Keep backward compatibility: existing `has_text`, `exit_code_zero`, `contains` conditions must continue working unchanged.

    **Part C: Edge-Aware Branch Routing**

    Modify the main `_run_workflow` execution loop to respect `sourceHandle` on edges when a conditional node returns a branch result. Currently the loop does NOT filter edges by sourceHandle -- all successors of a conditional execute regardless of branch.

    After a node completes successfully, check if it was a conditional node with a `branch` in its output. If so:
    1. Find all edges where `edge["source"] == node_id`
    2. Identify the matching branch: edges where `edge.get("sourceHandle") == branch_label`
    3. Identify non-matching edges: edges where `edge.get("sourceHandle")` is a non-None string and does not match `branch_label`
    4. Add all targets of non-matching edges to `skipped_nodes`
    5. Recursively mark their downstream nodes as skipped via `_mark_downstream_skipped`

    BACKWARD COMPATIBILITY: Edges where `sourceHandle` is missing or `None` are treated as unconditional (always-match) edges -- they execute regardless of the branch result. This ensures edges saved before this change (which lack `sourceHandle`) continue to work as pass-through connections. Only edges with an explicit non-None `sourceHandle` value participate in branch filtering.

    This change goes in the success path of the node execution loop (after `node_outputs[node_id] = output_msg`), before moving to the next node. It must work correctly with the existing `predecessors` and `successors` maps.

    **Part D: Model Updates**

    In `backend/app/models/workflow.py`:
    - Add `APPROVAL_GATE = "approval_gate"` to `WorkflowNodeType` enum
    - Add `PENDING_APPROVAL = "pending_approval"` to both `WorkflowExecutionStatus` and `NodeExecutionStatus` enums
    - Add `sourceHandle: Optional[str] = None` field to `WorkflowEdge` model (for edge-aware routing)

    IMPORTANT: Do NOT use `eval()` anywhere. The security concern from CONCERNS.md section 1.4 about arbitrary code execution applies directly here.

    **Part C: Approval Gate Node Executor**

    Add `"approval_gate": "_execute_approval_gate_node"` to `NODE_DISPATCHERS` dict.

    Implement `_execute_approval_gate_node` following the proven `TeamExecutionService._execute_human_in_loop` pattern (lines 589-661):
    1. Create a `threading.Event()` for this approval gate
    2. Read `timeout` from `node_config` (default 1800 seconds / 30 minutes)
    3. Read `execution_id` from `input_msg.metadata["_execution_id"]` (injected by Part B above)
    4. Register the approval event in the in-memory `_executions` dict under a new `_approval_events` sub-dict keyed by `(execution_id, node_id)`
    5. Persist to DB via `add_workflow_approval_state(execution_id, node_id, timeout)`
    6. Update node state to `pending_approval` via `_update_node_state(execution_id, node_id, "pending_approval")`. IMPORTANT: Verify that `_update_node_state` triggers SSE emission (it should -- the existing mechanism broadcasts node status changes to connected SSE clients). If `_update_node_state` does NOT emit SSE, add an explicit `_emit_sse_event(execution_id, {"node_id": node_id, "status": "pending_approval"})` call here. The frontend relies on this SSE event to show the approval modal (see 07-03 Plan, useWorkflowExecution).
    7. Call `approval_event.wait(timeout=timeout)`
    8. On timeout: update DB state to `timed_out`, raise RuntimeError
    9. On resume: check DB state -- if `rejected`, raise RuntimeError("Approval gate rejected"); if `approved`, return input_msg as pass-through

    Store approval events at class level: `_approval_events: Dict[tuple, threading.Event] = {}` with its own lock.

    NOTE: The `execution_id` is obtained from `input_msg.metadata["_execution_id"]` which is injected by Part B. If the key is missing, raise `RuntimeError("execution_id not available in input_msg.metadata -- _run_workflow must inject it")`.
