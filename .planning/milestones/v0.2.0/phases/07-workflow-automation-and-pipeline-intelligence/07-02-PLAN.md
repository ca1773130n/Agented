  <action>
    **Part A: Replace Agent Node Stub**

    The current `_execute_agent_node` (line ~810) returns a stub message: `[agent:{agent_id}] executed`. Replace it with real execution via `OrchestrationService`.

    New implementation for `_execute_agent_node`:
    1. Read `agent_id` or `trigger_id` from `node_config`
    2. Read `fallback_chain` from `node_config` (list of `{backend_id, account_id, priority}` entries) -- if not provided, use the default chain from `get_fallback_chain(entity_type="workflow", entity_id=workflow_id)`
    3. Read `routing_rules` from `node_config`:
       - `diff_size_threshold`: int (default 500 lines)
       - `small_diff_tier`: str (default "cheap") -- which chain entry priority tier for small diffs
       - `large_diff_tier`: str (default "expensive") -- which tier for large diffs
    4. Determine diff size from `input_msg.data.get("pr", {}).get("lines_changed", 0)` or `input_msg.data.get("diff_size", 0)`
    5. If routing_rules exist and diff size is available:
       - Below threshold: filter fallback chain to entries with `tier == small_diff_tier`
       - Above threshold: filter fallback chain to entries with `tier == large_diff_tier`
       - If filter produces empty list, use full chain (graceful fallback)
    6. Build a synthetic trigger dict from node_config (matching the format expected by `OrchestrationService.execute_with_fallback`)
    7. Call `OrchestrationService.execute_with_fallback(trigger_dict, message_text, event_data, "workflow")`
    8. Convert the execution result into a `WorkflowMessage` with the output text and metadata

    Handle the `CHAIN_EXHAUSTED` result from OrchestrationService: if all providers are rate-limited, raise RuntimeError so the node's retry/error_mode can handle it (per research Production Considerations).

    **Part B: Fallback Chain Config Model**

    In `backend/app/models/workflow.py`, add:
    ```python
    class FallbackChainEntry(BaseModel):
        backend_id: str
        account_id: str
        priority: int = 0
        tier: str = "default"  # "cheap", "expensive", "default"

    class RoutingRules(BaseModel):
        diff_size_threshold: int = Field(default=500, description="Line count threshold for cheap vs expensive model")
        small_diff_tier: str = Field(default="cheap")
        large_diff_tier: str = Field(default="expensive")

    class AgentNodeConfig(BaseModel):
        agent_id: Optional[str] = None
        trigger_id: Optional[str] = None
        fallback_chain: List[FallbackChainEntry] = Field(default_factory=list)
        routing_rules: Optional[RoutingRules] = None
    ```

    These models are for documentation and validation -- the actual node_config is a plain dict at runtime.

    **Part C: Enhanced Retry Policy Config**

    The existing `retry_max` and `retry_backoff_seconds` fields on `WorkflowNode` already implement WF-06 (per-node retry). Enhance the model to also support named backoff strategies:

    Add a `backoff_strategy` field to `WorkflowNode`:
    ```python
    backoff_strategy: str = Field(default="exponential", description="Backoff strategy: fixed, linear, exponential")
    ```

    Update the retry loop in `_run_workflow` to respect `backoff_strategy`:
    - `fixed`: delay = retry_backoff (constant)
    - `linear`: delay = retry_backoff * attempts
    - `exponential`: delay = retry_backoff * (2 ** (attempts - 1)) -- current behavior

    This is a minor enhancement to the existing retry code (line ~462) -- not a rewrite.

    AVOID: Do NOT re-implement fallback chain logic inside workflow nodes. Reuse `OrchestrationService.execute_with_fallback()` which already handles all edge cases (per research anti-pattern guidance).

    - `test_conditional_branch_skip_downstream` -- DAG with chain: conditional -> true_branch -> downstream_of_true. When false, both true_branch AND downstream_of_true must be skipped.

    - `test_approval_gate_timeout` -- Set timeout=1 second. Don't approve. Verify node times out and execution fails.
    - `test_approval_state_db_persistence` -- Verify approval state is written to and readable from `workflow_approval_states` table.

    - `test_retry_respects_max_attempts` -- Node that always fails. Verify exactly max_attempts+1 attempts, then failure.
