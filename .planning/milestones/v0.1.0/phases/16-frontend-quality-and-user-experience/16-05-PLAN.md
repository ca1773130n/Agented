<objective>
Add consistent error handling with toast notifications to all 21 async views, split into two groups based on whether the view already uses EntityLayout (which provides loading and error states natively) or not.

Purpose: Complete UX-01 coverage -- every async view surfaces API errors as toast notifications via handleApiError, and the 5 non-EntityLayout views are audited for loading/error state coverage. EntityLayout already provides loading spinners, error UI, and retry -- so EntityLayout views only need handleApiError added to their loadEntity functions for toast notification consistency, NOT separate LoadingState/ErrorState wrappers.

Depends on plan 16-01 for error-handler.ts (handleApiError).

Key insight: 16 of the 21 target views already use EntityLayout, which manages isLoading, loadError, loading spinner, error UI, and retry internally. Adding a separate isLoading ref + LoadingState + ErrorState template wrapper to these views would create nested loading spinners and duplicate error states. The correct approach for EntityLayout views is ONLY to add handleApiError calls inside their loadEntity functions for consistent toast notifications.
</objective>
