<task type="auto">
  <name>Task 1: Create ErrorBoundary component and centralized error handler</name>
  <files>frontend/src/components/base/ErrorBoundary.vue, frontend/src/services/api/error-handler.ts</files>
  <action>
    **ErrorBoundary.vue** — Create a Vue 3 component using `onErrorCaptured`:

    1. Define props: `fallbackTitle?: string` (default: 'Something went wrong')
    2. Reactive state: `hasError` (ref<boolean>), `errorMessage` (ref<string>), `recoveryKey` (ref<number>)
    3. `onErrorCaptured((err: Error, _instance, info: string)` handler that:
       - Sets `hasError.value = true`
       - Sets `errorMessage.value` to `err.message` with `(in ${info})` suffix
       - Returns `false` to stop propagation and prevent SPA crash
    4. `recover()` function that resets `hasError` and `errorMessage`, increments `recoveryKey` to force re-creation of child subtree
    5. Template:
       - When `hasError`: Show fallback UI with role="alert", the error icon (reuse the SVG pattern from ErrorState.vue), the title prop, the error message, and a "Try Again" button calling `recover()`
       - When no error: Render `<slot />` wrapped in a div with `:key="recoveryKey"` to enable subtree re-creation on recovery
    6. Style the fallback UI consistently with the existing ErrorState.vue design (same colors, spacing, dark theme variables)

    **error-handler.ts** — Create centralized error handler module:

    1. Import `ApiError` from `../api/client`
    2. Define `ErrorMapping` interface: `{ code: string; message: string; action: string }`
    3. Define `STATUS_MAP: Record<number, ErrorMapping>` covering these status codes:
       - 0: ERR-TIMEOUT, "Request timed out", "Check your connection and try again."
       - 401: ERR-401, "Unauthorized", "Check your API key in Settings."
       - 403: ERR-403, "Forbidden", "You do not have permission for this action."
       - 404: ERR-404, "Not found", "The resource may have been deleted. Return to the list."
       - 409: ERR-409, "Conflict", "The resource was modified. Refresh and try again."
       - 422: ERR-422, "Validation error", "Check your input and try again."
       - 429: ERR-429, "Rate limited", "Wait a moment and try again."
       - 500: ERR-500, "Server error", "The server encountered an error. Try again later."
       - 503: ERR-503, "Service unavailable", "The service is temporarily down. Try again shortly."
    4. Export `formatApiError(status: number, serverMessage?: string): string` that:
       - Looks up STATUS_MAP for the status code
       - If found: returns `"${mapping.message}${detail} (${mapping.code}). ${mapping.action}"` where detail includes serverMessage if it differs from `HTTP ${status}`
       - If not found: returns `serverMessage || "Unexpected error (ERR-${status}). Try again or contact support."`
    5. Export `handleApiError(error: unknown, showToast: (msg: string, type: string) => void, fallbackMessage?: string): string` that:
       - If error is ApiError: calls formatApiError and shows toast with 'error' type
       - If error is generic Error: shows `"${error.message} (ERR-UNKNOWN). Try again."` toast
       - Otherwise: shows fallbackMessage with ERR-UNKNOWN suffix
       - Always returns the formatted message string for callers that need it (e.g., to set local error state)
  </action>

  </done>

  <files>frontend/src/App.vue, frontend/src/main.ts, frontend/src/components/layout/AppSidebar.vue</files>
  <action>
    **App.vue changes:**

    1. Import `ErrorBoundary` from `./components/base/ErrorBoundary.vue`
    2. Import `handleApiError` from `./services/api/error-handler`
    3. Wrap the existing `<router-view>` with `<ErrorBoundary>`:
       ```
       <ErrorBoundary>
         <router-view />
       </ErrorBoundary>
       ```
       Keep the ErrorBoundary INSIDE the main content area only -- do NOT wrap the sidebar or toast container.
       The sidebar and toast container must remain OUTSIDE the ErrorBoundary so they degrade gracefully.

    4. Add sidebar loading coordination state:
       - `const sidebarLoading = ref(true)`
       - `const sidebarErrors = ref<Record<string, string | null>>({})` with keys: triggers, projects, products, teams, plugins, backends, version

    5. Replace the 7 independent `loadX()` calls in `onMounted` with a single `loadSidebarData()` function:
       - Create an array of `{ key: string, fn: () => Promise<void> }` for all 7 loaders
       - Wrap all in `Promise.allSettled(loaders.map(l => l.fn()))`
       - After settlement, iterate results and set `sidebarErrors.value[key]` to the rejection reason message (or null for success)
       - Set `sidebarLoading.value = false` after all settle

    6. Create a `retrySidebarSection(key: string)` function that:
       - Clears `sidebarErrors.value[key]`
       - Re-runs only that section's loader function
       - On failure, sets `sidebarErrors.value[key]` to the new error message

    7. Replace `console.warn('[Sidebar] Failed to load ...')` in each `loadX()` catch block with `handleApiError(err, showToast, 'Failed to load {entity}')`. Keep the catch block to prevent Promise.allSettled from missing individual errors, but surface the error to the user.

    8. Pass `sidebarLoading` and `sidebarErrors` as props to `AppSidebar`:
       - In App.vue, add `:sidebar-loading="sidebarLoading"` and `:sidebar-errors="sidebarErrors"` to the `<AppSidebar>` component usage.
       - In `AppSidebar.vue`, add two new props to the existing `withDefaults(defineProps<{...}>())`:
         ```typescript
         sidebarLoading?: boolean;
         sidebarErrors?: Record<string, string | null>;
         ```
         Default `sidebarLoading` to `false` and `sidebarErrors` to `{}`.
       - At the TOP of the sidebar template (inside the `<aside>` tag, before the first nav section), add a conditional loading indicator:
         ```vue
         <div v-if="sidebarLoading" class="sidebar-loading">
           <div class="sidebar-loading-spinner" />
           <span>Loading...</span>
         </div>
         ```
       - Style the spinner with a CSS animation keyframe (rotating border spinner, using `var(--accent-color)` for the active border segment) and position it centered within a small flex container. Keep styling consistent with the app's dark theme.
       - For sections with errors (non-null `sidebarErrors[key]`), show an inline error indicator next to the section header: a small warning icon and "Failed to load" text with a "Retry" button that emits a `retrySidebarSection` event with the section key. Add a `retrySidebarSection` emit to the component's `defineEmits`.
       - In App.vue, listen for `@retry-sidebar-section="retrySidebarSection"` on the `<AppSidebar>` component.

    9. Provide `retrySidebarSection` via `provide('retrySidebarSection', retrySidebarSection)` so child components can trigger section-specific retries.

    **main.ts changes:**

    1. Add `app.config.errorHandler` after `createApp(App)` and before `app.mount()`:
       ```typescript
       app.config.errorHandler = (err, instance, info) => {
         console.error('[Vue Error]', err, '\nComponent:', instance?.$options?.name || 'unknown', '\nInfo:', info);
       };
       ```
       This is defense-in-depth -- it catches errors that escape the ErrorBoundary (e.g., errors in event handlers) and logs them. It does NOT show a toast (the ErrorBoundary handles recovery UI for rendering errors; event handler errors should use try/catch + handleApiError at the call site).

    - Do NOT make sidebar loading block the router-view -- they load independently
    - Do NOT remove the individual `loadX()` functions -- keep them for single-section retry
