<objective>
Implement smart context chunking with merge/dedup (EXE-03) and conversation branching with tree-structured messages (EXE-04) as backend services, database layer, and API endpoints.

Purpose: Smart chunking enables processing large code contexts (>100KB) by splitting at semantic boundaries, running the bot against each chunk independently, and merging deduplicated results. Conversation branching allows forking from any point in an execution transcript to explore alternatives without losing the original thread.

Output: ChunkService, ConversationBranchService, conversation_messages and conversation_branches DB tables, and API endpoints for branch management.

Research basis: Code-aware recursive character splitting at 400-512 tokens per Pinecone/LangCopilot recommendations (08-RESEARCH.md Recommendation 3). Tree-structured conversation branching based on ContextBranch paper (arXiv:2512.13914, Guo 2025) achieving 58% context reduction and improved focus. Normalized string matching for deduplication per NAACL 2025 Findings (simpler approaches match sophisticated ones).
</objective>

       - `ChunkService` class with class-level methods.

       - Class constants following 08-RESEARCH.md Recommendation 3 (Pinecone/LangCopilot):
         ```python
         CODE_SEPARATORS = [
             "\nclass ",        # Class boundaries
             "\ndef ",          # Function boundaries
             "\nasync def ",   # Async function boundaries
             "\n\n\n",          # Triple newlines (major sections)
             "\n\n",            # Double newlines (paragraph/block boundaries)
             "\n",              # Single newlines (line boundaries)
         ]
         MAX_CHUNK_CHARS = 2000   # ~500 tokens at 4 chars/token
         OVERLAP_CHARS = 200      # ~10% overlap for context continuity
         SIZE_THRESHOLD = 102400  # 100KB threshold for triggering chunking
         ```

       - `chunk_code(content: str, max_chars: int = None) -> list[str]` method:
         a. If content is under SIZE_THRESHOLD, return [content] (no chunking needed)
         b. Try each separator in priority order (class > function > triple newline > etc.)
         c. Split content on the first separator that produces chunks under max_chars
         d. Add OVERLAP_CHARS of overlap between consecutive chunks for context continuity
         e. Prepend file import statements/headers to each chunk as a preamble (per 08-RESEARCH.md Pitfall 2: "Include import statements and file headers in chunk preambles")
         f. Fall back to character-limit splitting if no separator works
         g. Return list of chunk strings

       - `_extract_preamble(content: str) -> str` helper:
         a. Extract import lines and module-level docstrings from the beginning of content
         b. Return as a preamble string to prepend to each chunk

       - `deduplicate_findings(findings: list[str]) -> list[str]` method:
         a. Normalize each finding: lowercase, strip whitespace, remove line number references (r'\bline \d+\b' -> 'line N')
         b. Use normalized string as dedup key
         c. Return list of unique findings preserving first-occurrence order
         d. Per 08-RESEARCH.md Open Question 2: "Start with normalized string matching. Add semantic dedup only if testing reveals significant paraphrased duplicates."

       - `merge_chunk_results(chunk_results: list[dict]) -> dict` method:
         a. Collect all findings from each chunk's bot_output (split by newlines, filter empty)
         b. Deduplicate via deduplicate_findings()
         c. Compute stats: total_chunks, unique_findings count, duplicate_count
         d. Produce merged_output string with unique findings
         e. Return MergedChunkResults-shaped dict

    AVOID:
    - Do NOT use greedy quantifiers in splitting that could match the entire file
    - Do NOT process chunks synchronously in a request thread (per 08-RESEARCH.md Anti-Pattern: "Never process chunks sequentially in the request thread")
    - Do NOT implement semantic/embedding-based deduplication for v1 -- normalized string match is the starting approach
  </action>
