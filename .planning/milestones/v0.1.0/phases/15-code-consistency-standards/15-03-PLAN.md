<task type="auto">
  <name>Task 3: Standardize exception handling with severity levels</name>
  <files>backend/app/services/*.py</files>
  <action>
    **CON-05: Exception handling standardization**

    1. Audit all exception handling blocks in services:
       ```bash
       grep -rn 'except.*Exception' backend/app/services/ --include='*.py' | grep -v '__pycache__'
       ```

    2. For each `except` block, classify by severity level and apply the correct pattern:

       **LEVEL 1 -- Critical path (must log with exc_info, must return error or propagate):**
       Applies to: trigger execution, data persistence, user-facing operations.
       ```python
       try:
           result = critical_operation()
       except Exception as e:
           logger.error("Operation failed for %s: %s", context_id, e, exc_info=True)
           return {"error": "Operation failed"}, HTTPStatus.INTERNAL_SERVER_ERROR
       ```

       **LEVEL 2 -- Best-effort operation (log warning, continue):**
       Applies to: audit logging, metrics collection, notifications, cache updates.
       ```python
       try:
           save_audit_event(event)
       except Exception as e:
           logger.warning("Failed to save audit event: %s", e, exc_info=True)
           # Continue -- audit is not critical
       ```

       **LEVEL 3 -- Intentionally silenced (add explanatory comment):**
       Applies to: cleanup operations, optional feature detection, retry loops where failure is expected.
       ```python
       try:
           optional_cleanup()
       except Exception:
           pass  # Intentionally silenced: cleanup is best-effort, failure is harmless
       ```

    3. For each bare `except Exception: pass` block WITHOUT a comment, review the context:
       - If it is in a cleanup/teardown path -> add comment: `# Intentionally silenced: <reason>`
       - If it is in an operational path -> add `logger.warning()` or `logger.error()` as appropriate
       - Do NOT blindly add logging to all bare except blocks (per Pitfall 2 from research: "Treating all bare exception blocks as bugs rather than intentional design")

    4. Ensure `exc_info=True` is used consistently:
       - ALWAYS include `exc_info=True` in `logger.error()` calls inside except blocks
       - Include `exc_info=True` in `logger.warning()` calls where the traceback aids debugging
       - NEVER include `exc_info=True` in `logger.info()` or `logger.debug()` calls

    5. Also update the error returns in service methods to use ErrorResponse where they construct `{"error": ...}` dicts:
       ```python
       from app.models.common import ErrorResponse

       # Before:
       return {"error": "Agent not found"}, HTTPStatus.NOT_FOUND

       # After:
       return ErrorResponse(error="Agent not found").model_dump(), HTTPStatus.NOT_FOUND
       ```
