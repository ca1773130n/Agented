<objective>
Implement SQLite-backed execution queue with per-trigger concurrency caps and a background dispatcher thread, replacing the current fire-and-forget thread dispatch.

Purpose: Prevent resource exhaustion from concurrent bot executions by enforcing per-trigger concurrency limits, enable queue visibility in admin UI, and ensure pending executions survive server restart.
Output: ExecutionQueueService with background dispatcher, queue DB module, integration with ExecutionService dispatch flow, admin queue depth API.
Research basis: persist-queue (2025) SQLite-backed queue architecture; SQLite WAL mode for concurrent access; Nygard (2018) stability patterns.
</objective>

  <files>backend/app/services/execution_queue_service.py, backend/app/db/execution_queue.py, backend/app/db/schema.py, backend/app/db/migrations.py</files>
  <action>
    Based on persist-queue (2025) architecture and Agented conventions:

    1. Create `backend/app/db/execution_queue.py` with SQLite CRUD:
       - Table `execution_queue` with columns:
         - `id TEXT PRIMARY KEY` (prefixed ID: "qe-" + 6-char random, following existing ID conventions from db/ids.py)
         - `trigger_id TEXT NOT NULL`
         - `trigger_type TEXT NOT NULL` (webhook, github, schedule, manual)
         - `message_text TEXT`
         - `event_data TEXT` (JSON serialized event payload)
         - `status TEXT NOT NULL DEFAULT 'pending'` (pending, dispatching, completed, failed, cancelled)
         - `priority INTEGER DEFAULT 0` (for future use, FIFO within same priority)
         - `created_at TEXT NOT NULL`
         - `dispatched_at TEXT`
         - `completed_at TEXT`
       - Functions following existing db module patterns:
         - `enqueue_execution(trigger_id, trigger_type, message_text, event_data) -> str` (returns queue entry ID)
         - `get_pending_entries(limit=10) -> List[dict]` (ORDER BY priority DESC, created_at ASC -- FIFO)
         - `update_entry_status(entry_id, status)` with CAS
         - `count_active_for_trigger(trigger_id) -> int` (status IN ('dispatching'))
         - `get_queue_depth(trigger_id=None) -> int` (pending count, optionally filtered by trigger)
         - `get_queue_summary() -> List[dict]` (per-trigger pending/dispatching counts for admin)
         - `cancel_pending_entries(trigger_id=None) -> int` (bulk cancel pending entries)
         - `cleanup_completed_entries(max_age_hours=24) -> int` (remove old completed entries)

    2. Add `execution_queue` table to `backend/app/db/schema.py` in `create_fresh_schema()` and add migration (next sequential number after Plan 01's migration)

    3. Create `backend/app/services/execution_queue_service.py`:
       - Class `ExecutionQueueService` with `@classmethod` methods
       - `_dispatcher_thread: Optional[threading.Thread] = None`
       - `_stop_event = threading.Event()`
       - `_concurrency_caps: Dict[str, int] = {}` (per-trigger caps, default=1)

       - `start_dispatcher()`:
         - Start background daemon thread running `_dispatcher_loop()`
         - Called from `create_app()` during startup
         - Also restores any entries stuck in "dispatching" status back to "pending" (stale recovery)

       - `stop_dispatcher()`:
         - Set stop event, join thread with timeout

       - `enqueue(trigger_id, trigger_type, message_text, event_data=None) -> str`:
         - Insert into execution_queue table via `enqueue_execution()`
         - Return queue entry ID

       - `_dispatcher_loop()`:
         - While not stopped:
           - Get pending entries (batch of 10)
           - For each entry:
             - Check concurrency: `count_active_for_trigger(trigger_id)` vs cap
             - Check circuit breaker: `CircuitBreakerService.can_execute(backend_type)` (get backend_type from trigger config)
             - If both pass: update entry to "dispatching", spawn thread to run execution
             - If circuit breaker OPEN: leave in queue (will retry next poll cycle)
             - If concurrency cap reached: skip (will retry next poll cycle)
           - `_stop_event.wait(timeout=1.0)` -- 1-second poll interval

       - `_dispatch_entry(entry)`:
         - Call `ExecutionService.run_trigger()` with the entry's trigger and event data
         - On success: update entry to "completed"
         - On failure: update entry to "failed"
         - This runs in a daemon thread

       - `get_concurrency_cap(trigger_id) -> int`:
         - Check `_concurrency_caps` dict, default to 1
         - Future: read from trigger config (concurrency_cap column)

       - `set_concurrency_cap(trigger_id, cap)`:
         - Update in-memory cap

    4. Avoid these anti-patterns (from research):
       - Keep dispatcher transactions SHORT: SELECT pending, release connection, then dispatch
       - Do NOT hold DB connection while subprocess runs
       - Do NOT allow unbounded queue depth per trigger (cap at 100 pending entries; reject with 429 if exceeded)
