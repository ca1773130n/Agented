<objective>
Implement per-backend circuit breaker state machine and extend the existing retry mechanism to handle all transient execution failures (not just rate limits).

Purpose: Prevent cascading failures during AI backend outages by fast-failing new requests, and automatically recover from transient errors (502, 503, timeout) without user intervention.
Output: CircuitBreakerService with SQLite-persisted state, extended retry logic in ExecutionService, integration with OrchestrationService.
Research basis: Nygard (2018) "Release It!" circuit breaker pattern; Fowler (2014) three-state model; AWS "Exponential Backoff And Jitter" (2024) full jitter strategy.
</objective>

  <name>Task 1: Implement circuit breaker service with SQLite persistence</name>
  <files>backend/app/services/circuit_breaker_service.py, backend/app/db/circuit_breakers.py, backend/app/db/schema.py, backend/app/db/migrations.py</files>
  <action>
    Based on Nygard (2018) Chapter 5 circuit breaker pattern and PyBreaker API design:

    1. Create `backend/app/db/circuit_breakers.py` with SQLite CRUD:
       - Table `circuit_breakers` with columns: `backend_type TEXT PRIMARY KEY`, `state TEXT NOT NULL DEFAULT 'closed'`, `fail_count INTEGER DEFAULT 0`, `success_count INTEGER DEFAULT 0`, `last_failure_time REAL`, `updated_at TEXT`
       - Functions: `get_breaker_state(backend_type)`, `upsert_breaker_state(backend_type, state, fail_count, success_count, last_failure_time)`, `reset_all_breakers()` (for server startup)
       - Follow existing db module patterns: use `get_connection()` context manager from `app.db.connection`

    2. Add the `circuit_breakers` table to `backend/app/db/schema.py` in `create_fresh_schema()` and add a migration in `backend/app/db/migrations.py` (use next sequential migration number after the last existing one)

    3. Create `backend/app/services/circuit_breaker_service.py`:
       - Class `CircuitBreakerService` with `@classmethod` methods following existing service patterns
       - Three states: CLOSED (normal), OPEN (fast-fail), HALF_OPEN (trial)
       - Configurable per instance: `fail_max=5`, `reset_timeout=60` (seconds), `success_threshold=1`
       - Thread-safe via `threading.Lock()` -- one lock per backend type
       - In-memory state dict `_breakers: Dict[str, CircuitBreaker]` keyed by backend_type (claude, opencode, gemini, codex)
       - `get_breaker(backend_type) -> CircuitBreaker` -- lazy-initialize from SQLite state or defaults
       - `can_execute(backend_type) -> bool` -- check state, handle OPEN -> HALF_OPEN transition on timeout
       - `record_success(backend_type)` -- HALF_OPEN -> CLOSED on success_threshold met; CLOSED resets fail_count
       - `record_failure(backend_type, error)` -- increment fail_count, CLOSED -> OPEN on fail_max, HALF_OPEN -> OPEN immediately
       - `is_transient_error(error, exit_code) -> bool` -- classify errors: transient = rate limit stderr patterns, exit codes indicating 502/503/timeout/connection errors, subprocess.TimeoutExpired; non-transient = FileNotFoundError, exit code 1 with auth/prompt errors
       - `get_all_states() -> list` -- for admin visibility
       - `persist_state(backend_type)` -- write current in-memory state to SQLite
       - On startup: `restore_from_db()` loads persisted states; if recovery timeout has elapsed for an OPEN breaker, transition to CLOSED

    4. Avoid these anti-patterns (from research pitfalls):
       - Do NOT use a global circuit breaker -- one per backend_type
       - Do NOT trip on non-transient errors -- use excluded error classification
       - On server startup, reset OPEN breakers if recovery timeout has elapsed (handle stale OPEN state)
  </action>

  </verify>

  </done>

       - Before dispatching to a backend, call `CircuitBreakerService.can_execute(backend_type)`
       - If breaker is OPEN, skip this backend and try the next in fallback chain (existing fallback mechanism)
       - If all backends have OPEN breakers, return a structured error indicating circuit breaker rejection
       - After successful execution: `CircuitBreakerService.record_success(backend_type)`
       - After failed execution: check `CircuitBreakerService.is_transient_error()` first, only call `record_failure()` for transient errors

    2. Extend `ExecutionService.schedule_retry()` to handle transient execution failures beyond rate limits:
       - Currently only handles rate-limit retries. Extend the transient error detection in `_stream_pipe()` or post-execution status check
       - When a subprocess exits with a transient error pattern (connection refused, timeout, 502/503 in stderr), call `schedule_retry()` with appropriate backoff
       - Use the existing `pending_retries` table and retry mechanism -- no new retry infrastructure needed
       - The existing backoff formula `min(cooldown * 2^(attempt-1), MAX_RETRY_DELAY) + jitter` is correct per AWS (2024) -- keep it
       - Add classification: detect transient subprocess failures by checking stderr output for patterns like "connection refused", "timeout", "503", "502", "rate limit", "overloaded"
       - Cap retries at existing MAX_RETRY_ATTEMPTS (5)
