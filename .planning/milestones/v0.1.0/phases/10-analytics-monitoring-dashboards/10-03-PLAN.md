    **DB query:**

    Add a new function to `backend/app/db/analytics.py` (this file was created in Plan 10-01 which runs before this plan):

    `get_execution_time_patterns(trigger_id=None, days=90)` — Analyzes hour-of-day and day-of-week patterns from execution_logs. Returns two result sets:

    1. Hour patterns: `SELECT strftime('%H', started_at) as hour, COUNT(*) as total, SUM(CASE WHEN status='success' THEN 1 ELSE 0 END) as success, AVG(CASE WHEN status='success' THEN CAST((julianday(finished_at) - julianday(started_at)) * 86400000 AS INTEGER) END) as avg_duration_ms FROM execution_logs WHERE started_at >= ? GROUP BY hour ORDER BY hour`

    2. Day-of-week patterns: `SELECT strftime('%w', started_at) as day_of_week, COUNT(*) as total, SUM(CASE WHEN status='success' THEN 1 ELSE 0 END) as success, AVG(CASE WHEN status='success' THEN CAST((julianday(finished_at) - julianday(started_at)) * 86400000 AS INTEGER) END) as avg_duration_ms FROM execution_logs WHERE started_at >= ? GROUP BY day_of_week ORDER BY day_of_week`

    Per 10-RESEARCH.md Open Question 2: "optimal" = hour/day with highest success rate AND lowest average duration.

    **Service:**

    Create `backend/app/services/scheduling_suggestion_service.py`:

    ```python
    class SchedulingSuggestionService:
        MIN_EXECUTIONS_FOR_SUGGESTION = 10  # Need enough data for meaningful suggestion
    ```

    Methods:
    - `get_suggestions(cls, trigger_id=None)` — Calls `get_execution_time_patterns()`. If total executions < MIN_EXECUTIONS_FOR_SUGGESTION, returns empty suggestions with a message "Not enough execution history for suggestions (need at least 10)". Otherwise:
      a) For each hour: compute success_rate = success/total. Rank by success_rate DESC, avg_duration_ms ASC.
      b) For each day: same ranking.
      c) Return top 3 hours and top 2 days as suggestions.
      d) Each suggestion has: value (hour or day name), success_rate, avg_duration_ms, execution_count, rationale string (e.g., "Highest success rate (95%) with fastest average execution (2.3 min)").
    - `get_day_name(day_number)` — Maps 0-6 to Sunday-Saturday.

    **Pydantic models:**

    Add to `backend/app/models/analytics.py` (created in Plan 10-01):
    - `SchedulingSuggestion(BaseModel)`: type (Literal["hour","day"]), value (str), success_rate (float), avg_duration_ms (Optional[float]), execution_count (int), rationale (str)
    - `SchedulingSuggestionsResponse(BaseModel)`: suggestions (list[SchedulingSuggestion]), total_executions_analyzed (int), analysis_period_days (int), message (Optional[str])

    **API route:**

    Create `backend/app/routes/scheduling_suggestions.py`:

    ```python
    tag = Tag(name="scheduling-suggestions", description="Smart scheduling optimization")
    scheduling_bp = APIBlueprint("scheduling_suggestions", __name__, url_prefix="/admin", abp_tags=[tag])
    ```

    Endpoint: `GET /admin/analytics/scheduling-suggestions` — Query params: trigger_id (optional). Returns SchedulingSuggestionsResponse.

    Register in `backend/app/routes/__init__.py`.

    - `max_execution_time_seconds INTEGER` — Maximum execution time per run in seconds (NULL = no limit)
    - `max_monthly_runs INTEGER` — Maximum executions per calendar month (NULL = no limit)

    Add migration in `backend/app/db/migrations.py` to ALTER TABLE budget_limits ADD COLUMN for both fields.

    **DB layer:**

    Update `backend/app/db/budgets.py`:
    - Update `set_budget_limit()` to accept optional `max_execution_time_seconds` and `max_monthly_runs` parameters and persist them.
    - Add `get_monthly_run_count(entity_type, entity_id)` — `SELECT COUNT(*) FROM execution_logs WHERE trigger_id = ? AND started_at >= ? AND status != 'cancelled'` where `?` is the first day of the current month.
    - Update `get_budget_limit()` to include the two new columns in the returned dict.

    **Service layer:**

    Update `backend/app/services/budget_service.py`:
    - In `check_budget()` method (or wherever pre-execution budget checks happen), add a new check: if `max_monthly_runs` is set and `get_monthly_run_count() >= max_monthly_runs`, return a budget violation with reason "Monthly run limit exceeded (N/N)".
    - Add `check_execution_time_limit(entity_type, entity_id, elapsed_seconds)` — Returns True if `elapsed_seconds > max_execution_time_seconds` (when limit is set).

    **Execution service integration (with owner notification via health alerts):**

    In `backend/app/services/execution_service.py`, extend the existing `_budget_monitor()` thread (the one that polls every 30 seconds during execution):
    - Import `create_health_alert` from `app.db.health_alerts` (created in Plan 10-02).
    - After the existing cost budget check, add: compute elapsed time from execution start. Call `BudgetService.check_execution_time_limit()`. If exceeded, log a message "Execution time limit exceeded ({elapsed}s > {limit}s)", kill the process via `ProcessManager.cancel_graceful()` (per 10-RESEARCH.md Recommendation 4), AND create a health alert to notify the owner: `create_health_alert(alert_type='budget_exceeded', trigger_id=trigger_id, message=f'Execution cancelled: time limit exceeded ({elapsed}s > {limit}s)', details=json.dumps({'execution_id': execution_id, 'elapsed_seconds': elapsed, 'limit_seconds': limit}), severity='critical')`.
    - Update the pre-execution check (before subprocess.Popen) to call the monthly run count check. If exceeded, skip execution, set status to 'cancelled' with reason "Budget limit: monthly run count exceeded", AND create a health alert: `create_health_alert(alert_type='budget_exceeded', trigger_id=trigger_id, message=f'Execution blocked: monthly run limit exceeded ({current_count}/{max_runs})', details=json.dumps({'current_count': current_count, 'max_monthly_runs': max_runs}), severity='critical')`. Then return immediately.

    These health alerts use the infrastructure from Plan 10-02 and surface budget breach notifications in-app via the BotHealthDashboard (per roadmap success criterion #7: "the owner receives a notification").
