  <name>Task 1: Create analytics DB queries and service layer</name>
  <files>backend/app/db/analytics.py, backend/app/db/__init__.py, backend/app/models/analytics.py, backend/app/services/analytics_service.py</files>
  <action>
    Create `backend/app/db/analytics.py` with three SQL aggregation functions following the existing pattern from `budgets.py` `get_usage_aggregated_summary()`:

    1. `get_cost_analytics(entity_type, group_by, start_date, end_date)` — Aggregates `token_usage` table by entity (trigger/team/project), grouping by strftime period (day/week/month). Returns list of dicts with: entity_id, period, total_cost_usd, total_tokens (input_tokens + output_tokens). Use parameterized queries with `strftime()` format strings: day="%Y-%m-%d", week="%Y-W%W", month="%Y-%m". Filter by `entity_type` and optional date range on `recorded_at` column (indexed via `idx_token_usage_recorded`).

    2. `get_execution_analytics(group_by, start_date, end_date, trigger_id, team_id)` — Aggregates `execution_logs` table by time period. Returns list of dicts with: period, total_executions, success_count, failed_count, cancelled_count, avg_duration_ms, backend_type. Use `strftime()` on `started_at` column (indexed via `idx_execution_logs_started_at`). Filter with optional `trigger_id` (indexed via `idx_execution_logs_trigger_id`). Calculate avg_duration_ms as `AVG(CAST((julianday(finished_at) - julianday(started_at)) * 86400000 AS INTEGER))` for completed executions. Group by both period and backend_type.

    3. `get_effectiveness_analytics(trigger_id, start_date, end_date)` — Aggregates `pr_reviews` table. Returns dict with: total_reviews, accepted (review_status IN ('approved', 'fixed')), ignored (review_status = 'changes_requested' AND fixes_applied = 0), pending (review_status = 'pending'), acceptance_rate (accepted/total * 100, rounded to 1 decimal). Follow the existing `get_pr_review_stats()` pattern in triggers.py but add date filtering and per-period breakdown.

    4. `get_effectiveness_over_time(trigger_id, group_by, start_date, end_date)` — Time-series version of effectiveness. Groups by strftime period on `created_at`. Returns list of dicts with: period, total_reviews, accepted, acceptance_rate.

    Create `backend/app/models/analytics.py` with Pydantic v2 models:
    - `AnalyticsQuery(BaseModel)`: group_by (Literal["day","week","month"], default "day"), start_date (Optional[str]), end_date (Optional[str]), entity_type (Optional[str]), trigger_id (Optional[str]), team_id (Optional[str])
    - `CostAnalyticsResponse(BaseModel)`: data (list), period_count (int), total_cost (float)
    - `ExecutionAnalyticsResponse(BaseModel)`: data (list), period_count (int), total_executions (int)
    - `EffectivenessResponse(BaseModel)`: total_reviews (int), accepted (int), ignored (int), pending (int), acceptance_rate (float), over_time (list)

    Create `backend/app/services/analytics_service.py` with `AnalyticsService` class using @classmethod pattern (following MonitoringService). Methods:
    - `get_cost_summary(entity_type, group_by, start_date, end_date)` — Calls db function, computes total_cost from results. Defaults start_date to 30 days ago if not provided.
    - `get_execution_summary(group_by, start_date, end_date, trigger_id, team_id)` — Calls db function, computes total_executions. Defaults start_date to 30 days ago.
    - `get_effectiveness(trigger_id, group_by, start_date, end_date)` — Returns summary + over_time. Default trigger_id = None (all triggers with PR reviews).

    Add re-exports of the new db functions to `backend/app/db/__init__.py`.

    AVOID: Loading all rows into Python for aggregation — use SQL GROUP BY. AVOID: Complex window queries for this plan — simple GROUP BY + SUM/COUNT/AVG is sufficient (10-RESEARCH.md Anti-Patterns).
