    4. **Pydantic models** -- Create `backend/app/models/campaign.py`:
       - `CampaignCreate(name: str, trigger_id: str, repo_urls: list[str])` -- min 1 repo
       - `CampaignResponse(id, name, trigger_id, status, repo_urls, total_repos, completed_repos, failed_repos, started_at, finished_at, created_at)`
       - `CampaignExecutionResponse(campaign_id, execution_id, repo_url, status, started_at, finished_at, error_message)`
       - `CampaignPath(campaign_id: str)`

    5. **Campaign service** -- Create `backend/app/services/campaign_service.py`:
       - Class-level `_semaphore = threading.Semaphore(5)` (max 5 concurrent repo executions)

       - `start_campaign(name, trigger_id, repo_urls) -> str`:
         a. Create campaign in DB with status "running"
         b. Add campaign_execution entries for each repo (status "pending")
         c. For each repo, spawn a thread that calls `_execute_repo()`
         d. Spawn a monitor thread that waits for all repo threads to complete, then updates campaign status
         e. Return campaign_id

       - `_execute_repo(campaign_id, trigger_id, repo_url)`:
         a. Acquire semaphore (blocks if 5 already running)
         b. Update campaign_execution status to "running"
         c. Create a temporary project path for the repo
         d. Call `OrchestrationService.execute_with_fallback()` with the trigger and repo path
         e. Update campaign_execution with execution_id and status
         f. Release semaphore
         g. On exception: update status to "failed" with error_message, release semaphore

       - `_monitor_campaign(campaign_id, threads)`:
         a. Wait for all threads to complete (thread.join())
         b. Count completed/failed from campaign_executions
         c. Update campaign status to "completed" or "partial_failure"
         d. Call `NotificationService.on_execution_complete()` with campaign summary
         e. Audit log campaign completion

       - `get_campaign_results(campaign_id) -> dict`:
         a. Load campaign + all executions
         b. Group by repo_url with status, execution_id
         c. Return consolidated view

    Avoid: Per 11-RESEARCH.md Pitfall 5, do NOT spawn unlimited concurrent executions -- use the semaphore. Do NOT block the API request -- return campaign_id immediately and execute asynchronously. Each repo execution should be tracked independently so failures don't block others.
