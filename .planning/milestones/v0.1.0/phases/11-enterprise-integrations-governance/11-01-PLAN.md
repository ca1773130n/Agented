<objective>
Implement role-based access control (RBAC) with a decorator-based permission system and extend the existing audit trail to persist events to SQLite for queryable history.

Purpose: RBAC is the security foundation for all other enterprise features. Persistent audit trail is required by secrets vault (Plan 02), integrations (Plan 04), and governance compliance. Both must exist before downstream plans.

Output: `rbac_service.py` with `@require_role()` decorator, `audit_events` SQLite table with indexed queries, RBAC management routes, comprehensive tests.

Research basis: RBAC0 flat model (Sandhu et al. 1996), decorator pattern per Flask community conventions, event-sourced audit trail (Fowler 2005).
</objective>

    Based on RBAC0 model (Sandhu et al. 1996) and Flask decorator pattern from 11-RESEARCH.md:

    1. **DB schema** -- Add to `schema.py` `create_fresh_schema()` and add migration in `migrations.py`:
       ```sql
       CREATE TABLE IF NOT EXISTS user_roles (
           id TEXT PRIMARY KEY,
           api_key TEXT NOT NULL UNIQUE,
           label TEXT NOT NULL DEFAULT '',
           role TEXT NOT NULL DEFAULT 'viewer',
           created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
           updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
       );
       CREATE INDEX IF NOT EXISTS idx_user_roles_api_key ON user_roles(api_key);
       ```
       Roles are: `viewer`, `operator`, `editor`, `admin`.

    2. **ID generation** -- Add to `ids.py`:
       - `ROLE_ID_PREFIX = "role-"`, `ROLE_ID_LENGTH = 6`
       - `generate_role_id()` and `_get_unique_role_id(conn)` functions following existing pattern.

    3. **DB CRUD** -- Create `backend/app/db/rbac.py`:
       - `create_user_role(api_key, label, role) -> str` (returns role ID)
       - `get_role_for_api_key(api_key) -> Optional[str]` (returns role string or None)
       - `get_user_role(role_id) -> Optional[dict]`
       - `list_user_roles() -> list[dict]`
       - `update_user_role(role_id, label=None, role=None) -> bool`
       - `delete_user_role(role_id) -> bool`
       - Re-export all functions from `db/__init__.py`.

    4. **Pydantic models** -- Create `backend/app/models/rbac.py`:
       - `UserRoleCreate(api_key: str, label: str, role: str)` with role validation against allowed values
       - `UserRoleUpdate(label: Optional[str], role: Optional[str])`
       - `UserRoleResponse(id, api_key, label, role, created_at, updated_at)`
       - `RolePath(role_id: str)` for path parameter validation

    5. **RBAC service** -- Create `backend/app/services/rbac_service.py`:
       - Permission matrix as module-level constant:
         ```python
         ROLE_PERMISSIONS = {
             "viewer":   {"read"},
             "operator": {"read", "execute"},
             "editor":   {"read", "execute", "edit"},
             "admin":    {"read", "execute", "edit", "manage"},
         }
         ```
       - `require_role(*allowed_roles)` decorator that:
         a. Reads `X-API-Key` header from request
         b. If no API key header, falls back to checking if RBAC is enabled (when no roles exist in DB, RBAC is disabled and all requests pass -- graceful bootstrap)
         c. Looks up role via `get_role_for_api_key()`
         d. If role not in `allowed_roles`, logs denial via `AuditLogService.log(action="rbac.denied", ...)` and returns `{"error": "Insufficient permissions"}, 403`
         e. Otherwise calls the wrapped function
       - IMPORTANT: Apply `@require_role()` AFTER the route decorator (closer to function) to avoid breaking flask-openapi3 schema generation (see 11-RESEARCH.md anti-pattern)
       - `has_permission(api_key, permission) -> bool` utility for programmatic checks
       - `get_role_for_request(request) -> str` helper

    Avoid: Do NOT use Flask-RBAC extension (too heavyweight per research). Do NOT hardcode `if role == "admin"` checks scattered in code -- use the permission matrix.
