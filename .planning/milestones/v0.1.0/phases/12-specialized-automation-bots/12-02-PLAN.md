    **BOT-01: `.claude/skills/vulnerability-scan/INSTRUCTIONS.md`**
    - Title: "Dependency Vulnerability Scanner"
    - Steps: (1) Find package manifest files recursively (package.json, requirements.txt, pyproject.toml, Cargo.toml, go.mod, pom.xml, Gemfile.lock), (2) For each manifest, run `osv-scanner --format json --lockfile <file>` (if osv-scanner is available) OR fall back to `npm audit --json` / `pip-audit --format json` for npm/pip ecosystems, (3) Parse JSON output to extract vulnerability ID, severity, affected package, installed version, fixed version, (4) Cross-reference each finding with the package's role in the project (is it a direct dep or transitive?), (5) Score exploitability based on severity (CRITICAL > HIGH > MEDIUM > LOW), whether it's a direct dependency, and whether a fix version exists, (6) Produce a prioritized findings report as structured JSON with: package name, vulnerability ID, severity, exploitability score, installed version, fix version, and fix recommendation
    - Output format: JSON object with `findings` array and `summary` with counts by severity
    - Note in instructions: If `osv-scanner` is not found on PATH, log a warning and fall back to ecosystem-specific audit tools

    **BOT-02: `.claude/skills/code-tour/INSTRUCTIONS.md`**
    - Title: "Code Tour Generator"
    - Steps: (1) Scan the repository structure to identify project type (language, framework, build system), (2) Identify entry points (main files, app factories, CLI entry, test runners), (3) Map key abstractions (core classes, interfaces, types, domain models), (4) Trace data flow patterns (request lifecycle, data pipeline, state management), (5) Identify design patterns and architecture decisions (MVC, event-driven, etc.), (6) Document gotchas and non-obvious conventions (naming, file placement, implicit dependencies)
    - Output: Structured document with at minimum 5 labeled sections: "Entry Points", "Key Abstractions", "Data Flow", "Design Patterns", "Gotchas & Conventions". Each section must have subsections with file paths and explanations.

    - The PR comment must include: a summary of coverage gaps found, a table of untested functions with file paths, and specific test case suggestions
    - Include the bot identification footer: `---\n*Generated by Test Coverage Gap Detector via Agented*`

    **BOT-04: `.claude/skills/incident-postmortem/INSTRUCTIONS.md`**
    - Title: "Incident Postmortem Assistant"
    - Steps: (1) Parse the incident description from the input message to extract: time range, affected components/services, symptoms, (2) Search execution logs in the specified time range for errors, failures, and anomalies, (3) If a repo or PR context is provided, review related recent PRs and commits, (4) Construct a timeline of events from the gathered data, (5) Identify probable root cause(s) based on the timeline, (6) Assess impact (duration, affected users/systems, data loss), (7) Generate action items with owners and due dates
    - Output: Structured postmortem document with sections: "Summary", "Timeline", "Root Cause", "Impact", "Contributing Factors", "Action Items", "Lessons Learned"
    - Define "incident" as a time range + optional trigger/repo filter (per 12-RESEARCH.md Open Question 2)

    **BOT-05: `.claude/skills/generate-changelog/INSTRUCTIONS.md`**
    - Title: "Changelog Generator"
    - Steps: (1) List merged PRs since last release using `gh pr list --state merged --json number,title,labels,mergedAt,author --limit 100` (and `--base <branch>` if specified), (2) Parse each PR title using Conventional Commits spec (conventionalcommits.org v1.0.0): extract type prefix (feat, fix, docs, chore, refactor, perf, test, ci, build, style), scope, and description, (3) For non-conforming PR titles (no type prefix), classify using AI reasoning based on PR title content, (4) Group PRs by type: Features (feat), Bug Fixes (fix), Breaking Changes (titles containing BREAKING CHANGE or with `!` after type), Other (everything else), (5) Format as a CHANGELOG entry with version header, date, and grouped sections, (6) Optionally output as a GitHub Release draft body
    - Output: Markdown-formatted CHANGELOG entry following Keep a Changelog format
    - Per Conventional Commits spec (12-RESEARCH.md, Recommendation 4)

    **BOT-06: `.claude/skills/pr-summary/INSTRUCTIONS.md`**
    - Title: "PR Summary"
    - Steps: (1) Get the PR diff using `gh pr diff <number> --repo <repo>`, (2) Get PR metadata using `gh pr view <number> --repo <repo> --json title,body,files,additions,deletions`, (3) Analyze the diff to understand: what was changed (files, functions, components), why (infer from commit messages, PR title, PR body), and potential impact (breaking changes, performance, security), (4) Write a concise summary with 3 sections: "What Changed" (2-3 bullet points), "Why It Matters" (1-2 sentences), "Review Focus" (what reviewers should pay attention to), (5) Post the summary as a PR comment via `gh pr comment <number> --repo <repo> --body <body>`
    - Keep the prompt lightweight for the 60-second SLA (per 12-RESEARCH.md Pitfall 3) — focus on diff summary rather than deep code analysis
    - Include the bot identification footer: `---\n*Generated by PR Summary via Agented*`
    - Per Open Question 4 in 12-RESEARCH.md — always include a footer identifying the bot

    - Note: This bot's primary value is in the FTS5 search infrastructure (Plan 01), but the skill file guides Claude to interpret and present results meaningfully
