  <name>Task 1: Create useFormGuide composable with auto-discovery and help text extraction</name>
  <files>frontend/src/composables/useFormGuide.ts</files>
  <action>
    Create a Vue 3 composable that auto-discovers form fields and provides sequential navigation state.

    **FormField interface:**
    ```ts
    interface FormField {
      selector: string         // CSS selector for the .form-group element
      inputSelector: string    // CSS selector for the input/select/textarea within
      label: string            // Extracted from <label> element text
      helpText: string         // Extracted from help elements or data-tour-help
      isSubmit: boolean        // True for submit button field
      element: HTMLElement     // Reference to the .form-group element
    }
    ```

    **useFormGuide(containerRef: Ref<HTMLElement | null>) returns:**
    - `fields: Ref<FormField[]>` — discovered fields in DOM order, submit button appended last
    - `currentIndex: Ref<number>` — which field is currently highlighted (0-based)
    - `currentField: ComputedRef<FormField | null>` — the active field object
    - `isActive: Ref<boolean>` — whether form guidance mode is on
    - `totalFields: ComputedRef<number>` — fields.length
    - `activate()` — discover fields in container and set isActive=true, currentIndex=0
    - `deactivate()` — reset state, isActive=false
    - `nextField()` — advance currentIndex, return false if already at last
    - `prevField()` — go back, return false if at first
    - `reset()` — currentIndex=0 without deactivating

    **Auto-discovery logic (activate()):**
    1. Query `containerRef.value` for all `.form-group` elements
    2. For each .form-group, build a FormField:
       - `selector`: generate a unique CSS selector (use nth-of-type or data attributes if available)
       - `inputSelector`: find first `input`, `select`, or `textarea` within the .form-group. If none (e.g., checkbox group), use the .form-group itself
       - `label`: read the `<label>` element's `textContent.trim()`. Strip trailing asterisks/required markers. Fallback: "Field {n}"
       - `helpText`: Priority order:
         a. `data-tour-help` attribute on the .form-group — use its value directly
         b. `.form-help` or `.form-description` child element text
         c. `<small>` child element text (BackendDetailPage uses `<small>` for help text)
         d. Fallback: generate from label — "Enter the {label} for this form"
       - `isSubmit`: false for regular fields
    3. After all .form-group fields, find submit button:
       - Look in container for: `button[type="submit"]`, then `.btn-primary` within `.inline-form-actions` or form actions area, then `[data-tour="submit-btn"]`
       - If found, append a FormField with isSubmit=true, label="Save", helpText="Save your changes", and the button's selector as inputSelector
    4. Store result in `fields.value`

    **Selector generation:** For each .form-group, prefer `#id` on the input if available. If input has an id (e.g., `#account_name`), use `.form-group:has(#account_name)` as selector and `#account_name` as inputSelector. If no id, use index-based approach: `.form-group:nth-of-type({n})` scoped to the container.

    **Edge cases to handle:**
    - `.form-group.checkbox` — the input is a checkbox inside a label; use the whole .form-group as inputSelector
    - Conditional fields (v-if renders dynamically) — fields that aren't in DOM at discover time won't be found; this is acceptable since we discover at activate() time after the form is visible
    - Empty container — activate() sets fields to [] and isActive stays false

<task type="auto">
  <name>Task 2: Create TourFormGuide component with sequential field highlighting</name>
  <files>frontend/src/components/tour/TourFormGuide.vue</files>
  <action>
    Create a Vue 3 component that wraps useFormGuide and provides the bridge between form field discovery and the existing tour overlay system.

    **Props:**
    ```ts
    interface Props {
      active: boolean              // Whether form guidance mode is active
      containerSelector: string    // CSS selector for the form container (e.g., '.inline-account-form')
    }
    ```

    **Emits:**
    ```ts
    interface Emits {
      'field-change': [field: { target: string; message: string } | null]  // Emitted when current field changes
      'complete': []               // Emitted when user advances past last field (submit button)
    }
    ```

    **Implementation:**
    1. Use a `ref<HTMLElement | null>` for the container, resolved via `document.querySelector(props.containerSelector)` in a watch on `props.active`
    2. When `props.active` becomes true:
       a. Find container element via containerSelector
       b. Call `formGuide.activate()` to discover fields
       c. Emit `field-change` with the first field's inputSelector and helpText
    3. When `props.active` becomes false, call `formGuide.deactivate()` and emit `field-change` with null
    4. Expose `nextField()` and `prevField()` methods (called by parent):
       - `nextField()`: call formGuide.nextField(). If it returns false (was last field = submit button), emit 'complete'. Otherwise emit 'field-change' with new field data
       - `prevField()`: call formGuide.prevField(), emit 'field-change'
    5. Watch `formGuide.currentField` and emit `field-change` with `{ target: field.inputSelector, message: field.helpText }`

    **Template:** Renderless component — no visible DOM. Uses `<template>` with just a comment or empty div. All visual output is handled by TourOverlay/TourSpotlight/TourTooltip receiving the field-change data via the parent.

    **The field-change emit format matches TourOverlay's TargetLike interface:**
    ```ts
    { target: string; message: string }
    ```
    This allows App.vue to pass the emitted field data as `effectiveTarget` to TourOverlay, which already knows how to spotlight any CSS selector target and show the message in the tooltip.
