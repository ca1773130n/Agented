Purpose: Tour users should never be stuck staring at a spinner with no recovery path. Routes visited during the tour should be instant after the first navigation.
Output: Updated TourOverlay with timeout logic, updated useTourMachine with prefetchTourRoutes(), tests for the new states.

    Extend TourOverlay.vue with two resilience mechanisms:

    **1. Loading timeout (OB-40) — 5s spinner-to-fallback:**
    - Add a `loadingTimedOut` ref (boolean, default false)
    - Add a `routeLoadTimeout` ref holding a timeout ID
    - When `findTarget()` is called and no element is found, start a 5s timer: after 5000ms with still no target found, set `loadingTimedOut = true`
    - Clear the timer whenever an element IS found or when cleanup() is called
    - Add emit: `skip` and `retry` (retry already exists as implicit "re-run findTarget", surface as explicit event)
    - Add to template: a `v-if="!targetEl && loadingTimedOut"` fallback block replacing the spinner:
      ```html
      <div class="tour-timeout-fallback">
        <p class="fallback-text">This page is taking longer than expected.</p>
        <div class="fallback-actions">
          <button class="btn-fallback-skip" @click="$emit('skip')">Skip</button>
          <button class="btn-fallback-retry" @click="handleRetry">Retry</button>
        </div>
      </div>
      ```
    - `handleRetry()` function: sets `loadingTimedOut = false`, clears the timer, calls `findTarget()` again
    - The existing spinner (`v-if="!targetEl"`) should only show when `!loadingTimedOut`: change to `v-if="!targetEl && !loadingTimedOut"`
    - Style the fallback using CSS custom props: `var(--text-secondary)`, `var(--accent-cyan)` for retry, `var(--text-tertiary)` for skip

    **2. Scoped MutationObserver (OB-41) — scope to route root, 3s fallback:**
    - Current code observes `document.body` — change to observe `document.querySelector('#main-content') ?? document.body` (the route root element in App.vue template)
    - Add `elementNotFoundTimeout` ref (boolean, default false) and a separate 3s timer ref
    - After `nextTick() + 100ms` delay following a step change without finding target, start a 3s timer
    - If still not found after 3s, set `elementNotFoundTimeout = true` and show a different fallback:
      ```html
      <div v-if="!targetEl && elementNotFoundTimeout" class="tour-element-fallback">
        <p>We couldn't find <strong>{{ currentTargetName }}</strong>.</p>
        <div class="fallback-actions">
          <button @click="$emit('skip')">Skip</button>
          <button @click="handleElementRetry">Retry</button>
        </div>
      </div>
      ```
    - `currentTargetName` computed: derive a human-readable name from `props.step?.title` (e.g., "Workspace Directory input")
    - `handleElementRetry()`: sets `elementNotFoundTimeout = false`, clears timer, re-runs `findTarget()`
    - The 3s timer and 5s timer are independent — 3s is for "element not in DOM after route loaded", 5s is for "route itself is slow to load"
    - Reset both timers in `cleanup()` and whenever `props.step` or `props.effectiveTarget` changes

    **Avoid:**
    - Do NOT observe document.body (OB-41 explicitly requires scoped observer)
    - Do NOT show both fallbacks simultaneously — use priority: elementNotFoundTimeout takes precedence over loadingTimedOut when both could trigger
    - Do NOT use `any` types

    - `/settings` → `../../views/SettingsPage.vue`
    - `/backends/backend-claude` → `../../views/BackendDetailPage.vue`

    - `/backends/backend-gemini` → same
    - `/backends/backend-opencode` → same

    Since we cannot import view files directly from useTourMachine (would create coupling), the function should accept an optional array of prefetch functions OR be placed in a separate thin module. Simpler approach: add a standalone `prefetchTourRoutes()` function directly in `useTourMachine.ts` that calls:
    ```ts
    export async function prefetchTourRoutes(): Promise<void> {
      await Promise.allSettled([
        import('../views/SettingsPage.vue'),
        import('../views/BackendDetailPage.vue'),
      ])
    }
    ```
    Use `Promise.allSettled` (not `Promise.all`) so a single failure doesn't block. This is fire-and-forget; callers don't need to await it. Import paths from useTourMachine.ts: `../views/SettingsPage.vue` and `../views/BackendDetailPage.vue`.

    **In App.vue — call prefetchTourRoutes on tour start:**

      ```ts
        router.push(target)
      }
      ```
