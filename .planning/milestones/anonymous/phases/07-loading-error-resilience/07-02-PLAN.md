<task type="auto">
  <name>Task 2: Modal coordination — overlay yield when modal is open (OB-44)</name>
  <files>
    frontend/src/components/tour/TourOverlay.vue
    frontend/src/App.vue
  </files>
  <action>
    When a modal opens during a tour step (e.g., "Add Account" modal on the backends step), the tour overlay must step back to let the modal be interactive. The modal is already at z-index 1000 (App.vue line ~1271) — well below the tour overlay at 10000. Simply reducing the tour overlay's dim opacity while a modal is open makes the modal visually accessible and allows interaction.

    **Approach: prop-based modal coordination signal**

    The App.vue provides a `showToast` via inject/provide. Use the same pattern for a `tourModalOpen` signal.

    **In App.vue:**

    1. Add a `modalOpenDuringTour` ref (boolean):
       ```ts
       const modalOpenDuringTour = ref(false)
       ```

    2. Provide a setter so any modal component can signal it:
       ```ts
       provide('setTourModalOpen', (open: boolean) => {
         modalOpenDuringTour.value = open
       })
       ```

    3. Pass to TourOverlay as a new prop:
       ```html
       <TourOverlay
         ...
         :is-modal-open="modalOpenDuringTour"
         @next="tour.nextStep"
         @skip="tour.skipStep"
       />
       ```
       When the tour becomes inactive (tourActive becomes false), also reset modalOpenDuringTour to false:
       ```ts
       watch(tourActive, (active) => {
         if (!active) modalOpenDuringTour.value = false
       })
       ```

    **In TourOverlay.vue:**

    1. Add prop `isModalOpen: boolean` (default: false):
       ```ts
       const props = defineProps<{
         active: boolean
         step: StepLike | null
         effectiveTarget: TargetLike | null
         substepLabel: string | null
         stepNumber: number
         totalSteps: number
         isModalOpen?: boolean  // new
       }>()
       ```

    2. In the CSS, the `.tour-dim-fallback` overlay currently uses `var(--tour-overlay-dim)` (rgba(0,0,0,0.7)). Add a modifier class for modal mode:
       ```css
       .tour-dim-fallback.modal-open {
         opacity: 0.3;
       }
       ```
       Apply in template: `:class="['tour-dim-fallback', { 'modal-open': isModalOpen }]"`

    3. The TourSpotlight glow effect should also be dimmed when modal is open. Pass `isModalOpen` as a prop to TourSpotlight: add optional `reduced?: boolean` prop to TourSpotlight that applies a CSS class reducing glow opacity.

    **Wire modal open/close signal in the Add Account flow:**

    The backends step tour targets `[data-tour="add-account-btn"]`. When user clicks "Add Account", a modal opens. Look at how modals work in the existing codebase — search for modal usage in BackendDetailPage.vue or a shared modal component that has an open/close lifecycle.

    In any modal component that uses the standard modal pattern (look for `class="modal-overlay"` or `v-if="showModal"` pattern), inject `setTourModalOpen` and call it:
    ```ts
    const setTourModalOpen = inject<(open: boolean) => void>('setTourModalOpen', () => {})
    watch(() => showModal.value, (open) => setTourModalOpen(open))
    ```

    Find the modal in BackendDetailPage.vue (the "Add Account" form) and wire it. If the modal pattern is a teleported overlay, use `onMounted`/`onUnmounted` instead of watch.

    If there is no shared modal component (modals are inline v-if patterns), identify the specific modal in BackendDetailPage.vue and add the inject/watch there only.

    **Avoid:**
    - Do NOT make the tour overlay fully transparent when modal is open — keep 0.3 opacity for context
    - Do NOT use a global event bus — use the provide/inject pattern already established in App.vue
    - Do NOT break any existing tests by adding required props to TourOverlay (make isModalOpen optional with default false)
