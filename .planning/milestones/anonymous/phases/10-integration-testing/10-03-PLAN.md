       - `**/admin/backends/*` → `{ id: 'backend-claude', accounts: [] }` (no accounts by default)
    3. Export a `TourPage` class with helpers:
       - `constructor(page: Page)` — stores locators for `.tour-overlay`, `.tour-next-btn`, `.tour-skip-btn`, `.tour-step-counter`, `.tour-bottom-bar`
       - `async clickNext()` — clicks next button, waits for transition
       - `async clickSkip()` — clicks skip button, waits for transition
       - `async expectVisible()` — asserts overlay is visible
       - `async expectHidden()` — asserts overlay is not visible
       - `async expectStepText(text: string)` — asserts step counter contains text
       - `async startTour()` — navigates to `/?tour=start` and waits for overlay
    4. Extend base fixture to provide `tourPage: TourPage` as a fixture

    Note: The actual CSS class names for tour elements must match what TourOverlay.vue uses. Read the component to confirm selectors: `.tour-overlay`, `.tour-bottom-bar`, `.tour-next-btn`, `.tour-skip-btn`, `.tour-step-counter`. If the component uses different selectors, adjust the TourPage accordingly.

    **Step 2: Create `frontend/e2e/tests/tour-flow.spec.ts`**

    Write E2E tests using the tour fixture:

    **Test 1: "completes full tour with Next clicks"**
    - Navigate to app, start tour
    - Verify tour overlay appears with step counter
    - Click Next through each step, verifying step counter updates:
      - Welcome (STEP 1) -> Workspace (STEP 2) -> Claude (STEP 3) -> Codex (STEP 4) -> Gemini (STEP 5) -> OpenCode (STEP 6) -> Verification (STEP 7)
      - After final Next, tour overlay disappears (complete state)
    - Use `await expect(tourPage.overlay).toBeVisible()` (Playwright auto-waiting) — NEVER use `waitForTimeout()`

    **Test 2: "completes tour using Skip buttons"**
    - Start tour, click Skip on every skippable step
    - Verify tour reaches complete state
    - Verify tour overlay disappears after completion

    **Test 3: "keyboard navigation through tour"** (resolves Phase 6 deferred validation)

    - Press Enter to activate Next
    - Verify state advances
    - Press Tab to focus Skip button (if visible), press Space to activate
    - Verify Escape key does NOT dismiss the tour or skip the step (tour should be persistent)
    - Complete at least 2 steps via keyboard only

    IMPORTANT: Use Playwright's built-in auto-waiting (`expect(locator).toBeVisible()`, `locator.click()`) — NEVER `page.waitForTimeout()`. The existing codebase follows this pattern correctly.

    IMPORTANT: If the tour trigger mechanism is via `/?tour=start` query parameter, use `page.goto('/?tour=start')`. If it requires localStorage manipulation, use `page.evaluate()` to set the state. Check `useTour.ts` and app routing to determine the trigger mechanism.

<task type="auto">
  <name>Task 2: E2E persistence and accessibility tests</name>
  <files>frontend/e2e/tests/tour-persistence.spec.ts</files>
  <action>
    Create `frontend/e2e/tests/tour-persistence.spec.ts` using the tour fixture from Task 1.

    **Test 1: "tour state persists across page reload"** (resolves Phase 4 deferred validation)
    - Start tour, advance to step 2 or 3
    - Verify step counter shows the current step
    - Call `page.reload()`
    - Wait for page to load
    - Verify tour overlay reappears at the same step (not reset to idle or welcome)
    - The instance_id mock must return the same value before and after reload ('e2e-test-uuid')

    **Test 2: "tour state clears when instance-id changes"**
    - Start tour, advance to step 3
    - Override the instance-id mock to return a different UUID: `await page.route('**/health/instance-id', route => route.fulfill({ json: { instance_id: 'different-uuid' } }))`
    - Reload the page
    - Verify tour does NOT resume at step 3 — it should reset to idle (tour overlay hidden or starts from beginning)

    **Test 3: "reduced motion disables transitions"** (resolves Phase 8 deferred validation)
    - Before navigating, call `await page.emulateMedia({ reducedMotion: 'reduce' })`
    - Start tour
    - Verify tour overlay is visible
    - Check that CSS transitions are disabled: evaluate `getComputedStyle(element).transitionDuration` on a tour element — should be '0s' or 'none' if the app respects `prefers-reduced-motion`
    - If the app does not yet implement `prefers-reduced-motion` (this was Phase 8 work), the test should still pass but note the result. Write the test to check for the media query and assert accordingly — if transitions are still present, the test should document this as a known gap rather than fail.

    **Test 4: "focus trap within tour overlay"** (partial Phase 8 validation)
    - Start tour
    - Press Tab repeatedly, verify focus stays within the tour overlay controls (does not escape to the background app)
    - This tests whether the tour overlay traps focus as expected for accessibility
    - If focus trapping is not yet implemented (Phase 8 work), write the test to document the behavior rather than hard-fail

    IMPORTANT: Each test should be independent — do NOT rely on state from prior tests. The fixture's page.route() mocks are set up fresh per test.
