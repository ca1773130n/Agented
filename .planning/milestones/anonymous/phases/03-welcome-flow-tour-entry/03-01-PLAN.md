  <action>
    Replace the old `useTour` composable integration in App.vue with the new `useTourMachine` composable. This is a surgical replacement — the App.vue structure, layout, sidebar, toasts, health polling, and all non-tour code remain untouched.

    - Remove: `import { useTour } from './composables/useTour'`
    - Add: `import { useTourMachine } from './composables/useTourMachine'`
    - Change: `const tour = useTour()` -> `const tour = useTourMachine()`

    **Computed properties for TourOverlay binding:**
    The old `useTour` exposed `active`, `currentStep`, `effectiveTarget`, `substepLabel`, `displayStepNumber`, `totalSteps`. The new `useTourMachine` exposes `state`, `currentStep`, `isActive`, etc. Create a bridge layer in App.vue that maps machine state to TourOverlay props:

    1. Create a `tourActive` computed: machine `isActive` is true AND current route is NOT 'welcome'. The TourOverlay should NOT render while on the welcome page — the welcome page handles its own UI.

    2. Create a `tourStep` computed that maps the machine's current state to the StepLike interface that TourOverlay expects (`{ target, title, message, skippable }`). Define a STEP_META lookup object mapping state names to their display metadata:
       ```
       const TOUR_STEP_META: Record<string, { target: string; title: string; message: string; skippable: boolean; route: string; routeHash?: string }> = {
         'workspace': {
           target: '[data-tour="workspace-root"]',
           title: 'Workspace Directory',
           message: 'Set the root directory where repos will be cloned for your agent teams.',

           routeHash: '#general',
         },
         'backends.claude': {

           title: 'AI Backend Accounts',

           skippable: true,
           route: '/backends/backend-claude',
         },
         'backends.codex': {
           target: '[data-tour="add-account-btn"]',
           title: 'AI Backend Accounts',

           message: 'Confirm the harness plugin is installed and operational.',

         },
       }
       ```
       Note: This step metadata will be refined in Phase 4 (Core Step Content). For now, define the structural mapping so the overlay can render. Future phases will enrich these with per-field guidance and substep details.

    3. Create a `tourStepNumber` computed that derives the current step number (1-based) from the machine state. Map: workspace=1, backends.*=2, verification=3. Use a simple lookup.

    4. Set `totalSteps` to 3 (workspace, backends, verification — Phase 4 and Phase 9 will add more).

    5. Create a `tourSubstepLabel` computed: if current state is `backends.X`, return "Claude Code (1/4)" / "Codex CLI (2/4)" / "Gemini CLI (3/4)" / "OpenCode (4/4)". Otherwise null.

    6. The `effectiveTarget` prop on TourOverlay can be null for now — the step itself contains the target.

    **Route navigation on step change:**
    Add a `watch` on `tour.currentStep` that navigates to the correct route when the machine state changes. When machine enters 'workspace', navigate to '/settings#general'. When it enters 'backends.claude', navigate to '/backends/backend-claude', etc. Use `router.push()` with the route from TOUR_STEP_META. Only navigate if the current route differs from the target (avoid redundant navigations).

    **TourOverlay template binding update:**
    Change the TourOverlay props from old tour object to new computed values:
    ```
    <TourOverlay
      :active="tourActive"
      :step="tourStep"
      :effective-target="null"
      :substep-label="tourSubstepLabel"
      :step-number="tourStepNumber"
      :total-steps="totalTourSteps"
      @next="tour.nextStep"
      @skip="tour.skipStep"
    />
    ```

    **Tour start trigger:**
    Keep the existing `?tour=start` query param handling, but change to call the new machine methods:
    - In `onAuthenticated()` and the `watch` on `route.query.tour` and `onMounted`: when `tour=start` is detected, call `tour.startTour()` (which sends START event to machine, transitioning idle -> welcome), then immediately call `tour.nextStep()` (which sends NEXT, transitioning welcome -> workspace). This is because the welcome page has already been shown — the user is arriving from WelcomePage after key generation, so the machine should skip the welcome state and go straight to workspace.
    - Alternatively: when `tour=start` is detected, send START to go idle->welcome, then NEXT to go welcome->workspace. This two-step transition ensures the machine passes through the correct states.

    **Avoid:**
    - Do NOT touch any non-tour code in App.vue (sidebar, toasts, health, error handler, etc.)
    - Do NOT remove the old `useTour.ts` file — it will be cleaned up in a future phase
    - Do NOT change the TourOverlay component itself
    - The TOUR_STEP_META is intentionally minimal — Phase 4 will flesh out step content

    Wire WelcomePage.vue to the useTourMachine composable so the welcome page drives machine state transitions. The WelcomePage design is APPROVED — do not change the visual design, layout, CSS, or structure. Only add machine integration and transition choreography.

    **Machine integration:**
    1. Import `useTourMachine` composable:
       ```
       import { useTourMachine } from '../composables/useTourMachine'
       const tourMachine = useTourMachine()
       ```

    2. In `beginSetup()`: call `tourMachine.startTour()` to transition machine from idle -> welcome. This records that the tour has started. The function already sets `phase.value = 'keygen'` which shows the keygen card — keep that.

    3. In `continueToApp()`: Instead of directly calling `router.push({ path: '/', query: { tour: 'start' } })`, orchestrate the transition:
       a. Call `setApiKey(generatedKey.value)` (already there — keep it)
       b. Call `resetAuthGuard()` (already there — keep it)
       c. Call `tourMachine.nextStep()` to transition machine from welcome -> workspace
       d. Navigate to `/settings#general` (the workspace step route) directly instead of going through `/` with a query param. This eliminates the intermediate redirect through the dashboard:
          ```
          router.push({ path: '/settings', hash: '#general' })
          ```
       e. This is the critical change: instead of the user going Welcome -> / -> ?tour=start -> redirected to settings, they go Welcome -> /settings#general directly. The machine state change triggers TourOverlay to appear.

    **Cross-fade transition:**
    The WelcomePage already uses `<Transition name="phase-fade">` with 200-300ms CSS transitions. When the router navigates away from /welcome, Vue Router's component lifecycle will unmount WelcomePage. The App.vue layout switches from `welcome-fullscreen` to `app-body` layout. The TourOverlay renders because `tourActive` becomes true (machine is in 'workspace' state, route is not 'welcome').

    The potential "blank frame" happens if:
    - WelcomePage unmounts (welcome-fullscreen disappears)
    - App body renders with sidebar + main content
    - But the /settings page hasn't loaded yet, so TourOverlay has no target

    To prevent this, add a brief transition overlay. Create a `transitioning` ref:
    1. In `continueToApp()`, set `transitioning.value = true`
    2. Use `await nextTick()` + short timeout (50ms) to let the router start navigating
    3. The WelcomePage's root element should have an exit transition that fades out over 200ms

    The TourOverlay already handles missing targets with a loading spinner and MutationObserver (from Phase 2), so even if /settings takes a moment to render the workspace input, the overlay will show a spinner. This is acceptable — Phase 7 will add more sophisticated loading states.

    **Do NOT change:**
    - The welcome phase template (hero, bento grid, CTA)
    - The keygen phase template (generate button, key display, copy)
    - Any CSS or styling
    - The `onMounted` localStorage cleanup (this is correct for first-run)
    - The API call to `healthApi.setup('Admin')`

    **Important: The `onMounted` localStorage cleanup in WelcomePage currently removes `agented-tour-state` (old key). Also add removal of `agented-tour-machine-state` (new key from useTourMachine) to ensure a completely fresh start on the welcome page. The machine is re-initialized from idle when startTour() is called.**
